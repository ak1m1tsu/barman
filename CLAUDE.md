# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Commands

```bash
go build -o bin/bot ./cmd/bot/   # build
go test ./...                     # run all tests
go test ./internal/usecase/...   # run tests in one package
go test -run TestFetchGIF ./...  # run a single test by name
make lint                         # lint via Docker (golangci-lint)
make mock                         # regenerate mocks after interface changes
make hooks                        # install pre-commit hooks (run once after clone)
```

`make lint` runs inside Docker — Docker must be running. It checks `errcheck`, `govet`, `staticcheck`, and enforces `gofmt`/`goimports`.

## Git workflow

- Never commit directly to `main` — it is protected; all changes go through PRs.
- Before starting any task: `git checkout main && git pull`, then create a feature branch.
- Branch naming: `feat/<name>`, `fix/<name>`, `chore/<name>`, `refactor/<name>`.
- Commit after each completed logical step (one TODO item = one commit).

## New task trigger

When the user says **"Сделаем в новой задаче"** (or equivalent: "в новой задаче", "отдельной задачей"):
1. `git checkout main && git pull`
2. Merge / integrate the current feature branch if needed
3. `git checkout -b <new-branch>`
4. Implement changes
5. Write tests for every modified or new piece of business logic
6. Update `README.md` and `README.ru.md`

## Architecture

Dependencies must point strictly inward: `infrastructure → adapter → usecase → domain`.

- **domain** — entities and repository interfaces only; no external imports.
- **usecase** — business logic; depends only on domain interfaces, never on adapters or infrastructure.
- **adapter** — wires use cases to Discord (commands, handlers, repositories); no business logic here.
- **infrastructure** — external clients (HTTP, SQLite, config, logger); no business logic here.

Never import an outer layer from an inner one (e.g. no discordgo in `usecase/`, no HTTP clients in `domain/`).

### Composition root

`internal/app/app.go` is the single composition root. `New()` wires all dependencies (DB, repos, use cases, commands, handlers). `cmd/bot/main.go` only parses flags, loads config, calls `app.New` + `app.Run`, and handles OS signals. No wiring belongs in `main.go`.

### Slash commands vs event handlers

Two separate registration paths:

1. **Slash commands** — implement `NewXxxCommand() (*ApplicationCommand, Handler)`, register with `registry.Register(...)` in `app.go`. The `Registry` routes `InteractionApplicationCommand` events by name and logs every invocation automatically.

2. **Event handlers** — call `bot.Session.AddHandler(...)` directly in `app.go`. Used for non-command events (`GuildMemberAdd`, `MessageCreate`) and for interactive component flows.

### Dual-handler pattern for interactive commands

Some slash commands spawn buttons or modals (e.g. `/prefix`, `/autorole`). These require a second handler registered separately:

- `adapter/command/prefix.go` — handles the `/prefix` slash command, shows a message with buttons.
- `adapter/handler/prefix_interaction.go` — handles the subsequent button clicks (`InteractionMessageComponent`) and modal submits (`InteractionModalSubmit`).

Both are registered in `app.go`. The interaction handler is identified by `CustomID` constants defined in the handler file.

### Adding new functionality

- New business rule → `usecase/`
- New slash command → `adapter/command/`, register in `app.New`
- New event handler → `adapter/handler/`, add `bot.Session.AddHandler` in `app.New`
- New external API client → `infrastructure/`
- New persistence interface → `domain/`, implementation in `adapter/repository/sqlite/`

## Tests

- Every new `usecase/` file must have a corresponding `_test.go` in the same package.
- Cover: happy path, error propagation, all conditional branches.
- Use mocks from `mocks/` (generated by mockery). Run `make mock` after changing interfaces.
- Test package: `_test` suffix (e.g. `package reaction_test`). Assertions: `testify/assert` + `testify/require`.
- Mocks use `.EXPECT()` (mockery with-expecter mode). See `mocks/reaction/` for examples.

## Logging

- Every new piece of business logic must include structured logs via `logrus`.
- Log at key execution stages: entry point, branching decisions, errors, and completion.
- Always attach relevant context fields: `user_id`, `guild_id`, operation name, etc.
- Use `logrus.WithFields(...)` for structured entries; never use plain `fmt.Println`.
- Log level guidance: `Info` for normal flow milestones, `Warn` for recoverable anomalies, `Error` for failures.
- The custom JSON formatter (`internal/infrastructure/logger`) outputs fields in order: `time → level → msg → (other fields sorted)`.

## Linting

- Run `make lint` after every change before committing. Fix all reported issues before proceeding.
- `//nolint` is strictly forbidden. Code must always be written so that linters find no issues. If a linter fires, fix the root cause — never suppress it.

## Documentation

- Every exported identifier (type, function, method, variable, constant) must have a godoc comment that begins with the identifier's name.
- The comment must describe the purpose and any non-obvious implementation details clearly enough that a reader can understand the behaviour without reading the body.

## Config

`configs/config.yaml` (gitignored) is created from `configs/config.example.yaml`. Config is loaded by `internal/infrastructure/config` and passed to `app.New`. All handler timeouts come from `cfg.Timeouts.Handler` (default `10s` if unset). When adding new config fields, update `config.example.yaml` and both READMEs.

## README

Update both `README.md` and `README.ru.md` when:
- A new command or handler is added or removed
- The architecture changes (new package, removed package)
- Config fields are added or removed
